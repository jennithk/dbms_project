Information Model

The Information Model is the first step in database design. It defines what data we need to store based on real-world bookstore requirements. It focuses on entities like "Books" and "Authors" without worrying about technical coding yet.

This model ensures that business rules—such as "Every book must have an author"—are captured before we build the actual tables.

ER Model

The Entity-Relationship (ER) Model is a graphical blueprint. It uses rectangles for Entities (things), ovals for Attributes (details), and diamonds for Relationships (how things connect).

For example, a "Customer" entity has a "Places" relationship with an "Order" entity, showing exactly how data flows through the system.

DDL: Create & Alter

Data Definition Language (DDL) is the structural foundation of LIBRA. The CREATE command defines the table's "skeleton," specifying data types and rules for each column. As the bookstore grows, the ALTER command allows us to evolve the schema—adding new attributes like 'Genre' or 'E-book_Link' without losing existing data.

DDL: Drop & Truncate

Maintenance is key to a clean database. TRUNCATE is used to "empty the bucket"—it removes all records but keeps the table structure ready for future use. DROP is the "delete forever" command; it removes both the data and the table definition from the system catalog entirely.

DML: Data Entry

Data Manipulation Language (DML) represents the "daily life" of LIBRA. INSERT adds new arrivals to the shelves, UPDATE reflects price changes or stock refills, and DELETE removes records of books no longer in circulation. These commands power every transaction a customer makes.

Aggregate Operators

Raw data becomes "Business Intelligence" through aggregation. These functions perform math on multiple rows to return a single value. In LIBRA, we use SUM for total revenue, AVG for pricing strategy, and COUNT to track our total number of registered members and unique book titles.

Pattern Matching (LIKE)

Users rarely remember exact titles. The LIKE operator provides the search engine logic for LIBRA. Using the % wildcard (any characters) or _ (single character), we can find books even with partial info—essential for a user-friendly digital library experience.

Group By & Having

GROUP BY categorizes data into buckets, like counting books per genre. To filter these groups, we use HAVING. For example, if LIBRA wants to identify only the high-performing genres with more than 50 books in stock, the 'Having' clause acts as the final logic gate.

Nested Queries

A Nested Query (Subquery) is a query within a query. It allows LIBRA to solve complex, multi-step questions. For instance, to find books priced higher than the store average, the inner query first calculates the average, and the outer query then retrieves the matching books.

Integrity Constraints

Constraints are the "Guardrails" of the database, ensuring data remains professional and clean. NOT NULL prevents missing titles, UNIQUE stops duplicate ISBNs, and CHECK ensures prices are never negative. These rules maintain the reliability of the entire LIBRA system.

Database Views

A View is a "Virtual Table." It doesn't store data itself; it saves a query for frequent use. Views are used for security (hiding sensitive supplier costs) and simplicity, allowing us to create a 'Public_Catalog' that users can browse without seeing internal admin data.

Set Operations

Set operations combine results from different queries. UNION merges lists (like Sci-Fi and Fantasy) into one unique list. INTERSECT finds common ground between groups, and EXCEPT finds records present in one list but not the other—perfect for comparing inventory branches.

Relational Algebra

Relational Algebra is the mathematical foundation of RDBMS. It defines how data is manipulated using operators.

Fundamental Operations

  • Selection (sigma): Filters tuples based on a condition.
    sigma_{Genre='Manga'}(Books)
  • Projection (pi): Selects specific attributes (columns).
    pi_{Title, Price}(Books)
  • Rename (rho): Changes the name of attributes or relations.
  • Natural Join (bowtie): Merges tables based on common attributes.

This sequence tells the database: "First, look at the Books table, find those over 300, then show me only their titles."
